## 1. 專案初始問題與背景

一開始討論的出發點是：朋友做了一個「生成 IG 文案」的網站，使用者想理解這類網站到底是怎麼運作的。

初步結論如下：

- 這類網站通常不是使用 ChatGPT / Claude 網頁版的聊天額度。
- 正常做法是網站後端串接 OpenAI、Anthropic Claude、Gemini 等 AI API。
- 使用者在前端輸入需求，後端整理成 prompt，再呼叫 AI API 回傳文案。
- API 通常是另外計費，依 input token 與 output token 用量扣款。
- 若朋友提到「儲值 10 美金用到現在」，代表他很可能是使用 AI API 的 prepaid credits 或 API billing，而非聊天訂閱額度。

典型流程：

```text
使用者輸入商品、照片描述、語氣、Hashtag 要求
→ 前端送到後端
→ 後端整理 prompt
→ 呼叫 Claude / OpenAI / Gemini API
→ AI 回傳文案
→ 前端顯示結果
```

重要技術注意事項：

- API Key 不應放在前端，否則可能被瀏覽器 DevTools 看到並遭偷刷。
- 正確架構應是前端呼叫自己的後端，由後端安全地呼叫 AI API。
- 這類文案生成任務通常 token 消耗不高，因此 10 美金可以用很久是合理的。

## 2. 單純 IG 文案生成器的效益判斷

單純「輸入主題 → 生成 IG 文案」的產品效益有限。

原因：

- 使用者可以直接問 ChatGPT、Claude、Gemini 生成文案。
- 如果網站只是把 AI 包一層 UI，本質上可取代性很高。
- 真正有價值的不是「能不能生成」，而是「是否把生成包裝成一套穩定、低摩擦、可重複使用的工作流」。

低價值版本：

```text
輸入主題 → 產生一篇 IG 文案
```

高價值版本：

```text
上傳照片 / 輸入主題
→ 選擇帳號定位
→ 選擇貼文目的
→ 選擇語氣風格
→ 產生文案
→ 產生 Hashtag
→ 產生限動文字
→ 產生 Reels 標題
→ 儲存歷史紀錄
→ 沿用品牌風格
```

因此，「文案生成器」若要提升價值，必須做到：

1. 輸入項目多樣化。
2. 輸出項目多樣化。
3. 流程固定化。
4. 品牌風格記憶。
5. 歷史紀錄與素材管理。
6. 圖片分析與社群內容企劃。

## 3. 從 IG 文案生成器延伸出的新產品構想

使用者後續提出一個更有特色的想法：

> 將各家廠牌、不同系列、不同角色的模型輸入資料庫，做成一個模型攝影靈感產生器。使用者可以隨機抽幾隻模型，系統推薦拍攝主題、搞笑內容、角色組合或創作靈感。概念類似「不知道吃什麼就轉輪盤」。

這個方向比單純 IG 文案生成器更有產品差異。

因為它不是單純賣「AI 會寫字」，而是解決模型玩家與模型攝影者的實際痛點：

- 今天不知道拍哪一隻模型。
- 不知道不同模型怎麼搭配。
- 不知道該拍什麼主題。
- 不知道怎麼設計搞笑短劇。
- 不知道怎麼讓收藏品重新被使用。
- 不知道如何把模型拍攝內容轉成社群貼文。

產品本質：

```text
模型資料庫
+ 收藏庫
+ 靈感輪盤
+ 搭配推薦器
+ 拍攝主題產生器
+ AI 社群內容生成器
```

## 4. 建議產品定位

暫定產品名稱：

> FigureShot Lab｜模型攝影靈感與收藏搭配平台

一句話定位：

> 幫模型玩家從自己的收藏中產生拍攝靈感、角色搭配、搞笑劇情與社群內容，並逐步累積模型收藏與創作偏好資料。

更成熟的定位可寫成：

> 一個結合模型收藏管理、靈感輪盤、拍攝主題推薦與 AI 社群內容生成的平台，協助全球模型玩家與攝影創作者突破創作卡關，並讓收藏品產生更多拍攝與分享價值。

市場策略暫定：

- 採 **Global-first** 思維規劃，國際市場為主要成長方向。
- 介面與資料結構至少支援英文與繁體中文。
- 台灣市場可作為早期測試場域，但不把產品限制為台灣在地工具。
- 後續品牌、零售商、聯盟行銷與社群合作，均應預留跨國擴展能力。

## 6. 產品核心價值

本產品真正的價值不在於「AI 生成」，而在於：

1. **降低創作門檻**  
   使用者不用懂 prompt，也能透過表單、輪盤、標籤快速得到創作靈感。

2. **提高收藏使用率**  
   讓使用者從已有模型中找到新的拍攝主題，而不是收藏後只放著。

3. **增加社群分享內容**  
   將模型、場景、情緒、任務組合成可以發 IG、Threads、Reels 的內容。

4. **建立模型攝影工作流**  
   從「選模型 → 抽主題 → 生成企劃 → 拍攝建議 → 文案輸出」形成完整流程。

5. **累積匿名化市場洞察**  
   長期可分析玩家收藏偏好、拍攝偏好、願望清單與角色搭配趨勢。

## 7. 第一版 MVP 建議範圍

野心可以很大，但第一版要控制範圍。

第一版不建議直接做「全世界模型資料庫」，而是先做：

> 使用者自建收藏庫 + 靈感輪盤 + AI 拍攝企劃 + IG 文案生成

### MVP 核心流程

```text
使用者新增自己的模型
→ 為模型加標籤
→ 系統隨機抽 1～3 隻模型
→ 抽場景、情緒、任務
→ 產生拍攝主題
→ AI 生成拍攝企劃與文案
→ 使用者儲存或分享結果
```

## 10. 技術架構初步建議

### 10.0 產品介面與 UX 定位

視覺方向暫定為 **Cinematic Editorial Workspace**，不採一般企業 SaaS 儀表板的制式感。

核心設計原則：

- 首頁以大型模型攝影 Showcase 破題，讓使用者先感受到創作氛圍。
- 首頁核心問題可使用：`What do you want to shoot today?`
- 兼具攝影作品集的沉浸感與創作工具的可操作性。
- 工作區聚焦在「模型選擇、輪盤、靈感結果、生成與儲存」，避免過度堆疊資訊。
- 預留英文／繁體中文語言切換。
- 手機版需保留輪盤與快速生成的主要體驗，桌面版則強調收藏管理與編輯工作區。
- Wireframe 階段需先驗證首頁 Showcase 是否會干擾核心 CTA。



### 10.1 前端

建議使用：

- Vue 3
- Vite
- TypeScript 或 JavaScript
- Pinia
- Vue Router
- Tailwind CSS
- Element Plus / Naive UI

主要頁面：

- 首頁 / Landing Page
- 使用者登入頁
- 收藏庫列表頁
- 新增 / 編輯模型頁
- 模型詳情頁
- 靈感輪盤頁
- 生成結果頁
- 歷史紀錄頁
- 個人設定頁
- 未來趨勢儀表板頁

### 10.2 後端 / BaaS

> 尚未定案：Supabase、Firebase、Spring Boot 皆屬候選方案；在正式開發前必須完成 ADR（Architecture Decision Record），避免由後續 AI 自行替專案做隱性決策。

初期可選：

- Supabase
- Firebase
- 或 Spring Boot + PostgreSQL / MySQL

以快速 MVP 來說，Supabase / Firebase 會比較快。

若使用者想練習完整前後端，也可以用：

```text
Vue 3 前端
+ Spring Boot 後端
+ PostgreSQL / MySQL
+ OpenAI / Claude API
```

### 10.3 AI API

可接：

- OpenAI API

## 13. 「免費換資料」概念的修正

使用者提出的概念：

> 免費讓收藏家操作，累積數據給我們，再把資料賣給品牌公司。

這個方向抓到核心，但講法過於粗糙，會涉及隱私、信任與法規問題。

成熟說法應修正為：

> 我們提供模型玩家收藏管理與拍攝靈感工具，協助他們提升收藏使用率與社群創作效率；平台在使用者授權與資料去識別化後，彙整模型收藏、願望清單與創作偏好，形成市場趨勢洞察，供品牌、代理商與通路作為新品企劃、活動行銷與庫存策略參考。

核心原則：

```text
不是免費換資料
而是工具價值換信任
不是賣個資
而是賣去識別化的趨勢洞察
不是一開始找大品牌
而是先累積玩家使用行為與社群影響力
```

## 14. 資料倫理與隱私注意事項

收藏資料其實具有一定敏感性，例如：

- 使用者收藏了哪些模型。
- 使用者可能的消費能力。
- 使用者喜歡哪些 IP。
- 使用者願望清單。
- 使用者拍攝與分享偏好。

因此平台必須注意：

1. 使用者要知道平台蒐集哪些資料。
2. 使用者要能選擇是否參與匿名趨勢分析。
3. 資料應去識別化與彙整化。
4. 不應出售個別使用者收藏明細。
5. 不應把個別使用者行為交給品牌。
6. 要提供刪除帳號與資料的機制。
7. 隱私政策要清楚。
8. 使用者上傳照片的使用權限要清楚。

可接受的 B 端資料形式：

```text
本季 1/12 可動模型在模型攝影族群中的使用率上升
辦公室日常題材與反差搞笑題材互動最高
某類配件需求明顯增加
願望清單中某些系列成長快速
```

不可接受的形式：

```text
某某使用者收藏了哪些模型
某某使用者想買哪些商品
某某使用者偏好哪些 IP
某某使用者的消費能力推測
```

## 20D. 推薦引擎與 AI 責任邊界

### 20D.1 第一版推薦流程

```text
候選收藏集合
→ 套用使用者鎖定條件
→ 排除近期重複與不相容條件
→ 計算搭配分數
→ 加入可控隨機性
→ 選出模型、場景、情緒與任務
→ 規則層生成結構化企劃骨架
→ AI 將骨架轉為自然語言、腳本與社群內容
```

### 20D.2 建議評分因子

| 因子 | 可能影響 |
|---|---|
| 比例與尺寸相容性 | 提高可實際拍攝的可行性 |
| 同作品／同系列關係 | 提高原作關聯性 |
| 跨作品反差 | 提高創意與搞笑潛力 |
| 顏色與視覺協調 | 提高畫面一致性 |
| 場景適配度 | 避免模型與場景完全不相容 |
| 最近使用頻率 | 降低短期重複 |
| 冷門收藏曝光 | 讓長期未拍模型重新被使用 |
| 使用者偏好 | 逐步個人化推薦 |
| 拍攝難度 | 依使用者選擇控制可執行性 |

第一版不需要機器學習，可先使用規則加權與隨機抽樣。當資料量足夠後，再評估 Collaborative Filtering、Content-Based Recommendation 或 Learning-to-Rank。

AI 負責：

- 把結構化結果寫成自然語言。
- 生成拍攝腳本、對白、文案與變體。

規則引擎負責：

- 候選篩選。
- 搭配可行性。
- 推薦分數。
- 重複控制。
- 可解釋的推薦原因。

## 20E. 使用者角色與權限模型

| 角色 | 主要權限 |
|---|---|
| 訪客 | 瀏覽公開頁、試玩受限輪盤 |

正式開發前需補一份 RBAC 權限矩陣，至少定義新增、讀取、修改、刪除、發布、匯出與報表權限。

### 20F.3 品牌洞察的有效性限制

平台資料不能直接等同整體市場。正式對外提供趨勢洞察前，必須：

- 公開樣本期間、樣本數與市場分布。
- 排除機器人、灌票與單一重度使用者的過度影響。
- 區分收藏、願望清單、點擊與實際購買。
- 標註 self-selection bias。
- 避免將生成次數直接解讀為市場需求。
- 僅在樣本數達到最低門檻時發布細分報表。
- 對品牌清楚揭露資料限制與推論範圍。

## 20G. IP、內容安全與隱私治理

### 20G.1 IP 與圖片

- 初期以使用者自行上傳的收藏照片為主。
- 不大量抓取或重新散布未授權官方商品圖。
- 公共資料僅保存必要的文字識別資訊與可合法使用的素材。
- 使用者上傳時需同意其擁有上傳與展示權利。
- 需提供侵權申訴、DMCA 類流程、下架與回應紀錄。
- 品牌、角色與系列名稱的使用應以辨識與索引用途為限，不造成官方合作誤認。

### 20G.2 內容安全

- 建立公開內容檢舉與審核機制。
- 定義成人向模型、暴力、仇恨、騷擾與違法內容規範。
- 圖片上傳需限制格式、容量與尺寸，並執行惡意檔案檢查。
- AI 輸出應具備內容安全檢查與失敗降級處理。
- 公開分享前可加入人工或自動審核。

### 20G.3 隱私與 Consent

- 分開記錄必要服務資料與可選的匿名趨勢分析資料。
- 使用者可以撤回分析同意。
- 使用者可以匯出與刪除個人資料。
- 不向品牌提供可識別的個人收藏明細。
- 資料保留期限、刪除排程與備份刪除規則需明確。

## 20H. 成本模型與防濫用

### 20H.1 應估算的單位成本

- 每次純文字生成成本。
- 每次圖片理解成本。
- 圖片儲存與 CDN 成本。
- DB、Log、備份與分析成本。
- 免費會員月均成本。
- Pro 會員月均成本與毛利。
- 1,000、10,000、100,000 次月生成量的成本情境。
- AI 重試、失敗與多版本生成成本。

### 20H.2 防濫用措施

- API Rate Limit。
- 帳號與 IP 配額。
- CAPTCHA 或機器人防護。
- AI 生成次數與圖片分析配額。
- 異常流量偵測。
- Secret Rotation 與稽核 Log。
- 失敗重試上限與 Circuit Breaker。
- 公開分享連結防垃圾內容與大量產生。

定價必須在完成成本模型與市場驗證後才可定案；目前 NT$ 或美元價格都僅能視為假設。

## 20J. MVP 驗收與 Pivot 條件

### MVP 建議最低驗收

- 使用者可以完成註冊或使用試玩模式。
- 可以新增、編輯、刪除自己的收藏。
- 可以從收藏中抽取 1～3 隻模型。
- 可以鎖定模型並重新抽取其他條件。
- 可以取得結構化拍攝企劃。
- 可以呼叫 AI 產生至少一種腳本與社群文案。
- 可以儲存、重新開啟與分享靈感結果。
- 所有核心事件均有埋點。
- 使用者可查看與管理 Consent。
- API Key 不暴露於前端。

### 可能的 Pivot 訊號

- 玩家願意試玩，但不願建立收藏：轉向無帳號快速輪盤或匯入式收藏。
- 收藏建立率高，但輪盤回訪低：強化收藏管理與展示，而非強推 AI。
- 輪盤使用率高，但 AI 文案採用率低：降低 AI 比重，強化模板與攝影企劃。
- 分享率高但付費意願低：優先發展品牌活動、導購與贊助。
- B 端有興趣但 C 端樣本不足：先做聯名活動與限定挑戰，而非販售趨勢報告。

## 21. 最終總結

本專案從「AI 生成 IG 文案網站」的討論，延伸出更有差異化的產品方向：

> **模型攝影靈感與收藏搭配平台。**

核心價值不是單純生成文案，而是將模型收藏、標籤系統、靈感輪盤、搭配推薦、拍攝企劃與 AI 文案生成整合成完整工作流。

初期應專注於：

```text
讓模型玩家願意新增自己的收藏
讓他們願意反覆使用靈感輪盤
讓生成結果有趣到願意保存或分享
```

中期應專注於：

```text
建立社群活動
累積分享內容
分析收藏與拍攝偏好
建立模型攝影圈的使用者黏著度
```

長期可發展為：

```text
Pro 訂閱工具
導購分潤平台
品牌拍攝挑戰合作
模型市場趨勢洞察報告
品牌新品企劃與活動顧問服務
```

最重要的商業修正是：

> 不是「免費讓玩家用，然後賣他們的資料」，而是「提供有價值的工具，取得使用者信任與授權，將去識別化、彙整後的市場洞察提供給品牌」。

這樣才是可長期經營、可對外合作、也比較不會踩隱私與信任風險的方向。
